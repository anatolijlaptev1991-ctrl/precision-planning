# Changelog laptev-plan

## v5.1.1 (2026-07-19)

- Начало SKILL.md переписано в двух слоях: простое объяснение для человека и строгий контракт для автономного агента.
- Добавлен словарь терминов: LSH, атомарный шаг, evidence, артефакт, risk tag, rollback, readiness gate и executor constraints.
- Уточнены нормативные слова `ОБЯЗАН`, `ЗАПРЕЩЕНО`, `МОЖЕТ` и приоритет машинного контракта над пояснительным текстом.
- Функциональная модель v5.1.0 не изменена.

## v5.1.0 (2026-07-17)

Синтез по итогам аудита GPT-5.6 (72/100 → цель 90/100) и разбора Kimi. Исходная база — v5.0 (2588 строк, монолит).

### Главные изменения

**1. Модель-независимость восстановлена (аудит П.1 — главный регресс v5.0)**
* Удалены `executor_class` (`small_local_llm`/`large_llm`/`hybrid`/`shell_only`), имена моделей (Qwen), классификация по числу параметров, эскалация `big_model`.
* Введён `executor_constraints`: измеримые ограничения (контекст, вывод, структурированный вывод, действия на шаг). Пороги атомарности выводятся из ограничений, а не назначаются по классу.
* `escalation: big_model` заменена на `escalate_constraints` (тот же контракт исполнителю с более широкими ограничениями).
* Добавлен запрет: назначать пороги по имени/классу/размеру модели; вшивать списки моделей в спецификацию.

**2. Детекция исполнителя (уточнение пользователя про Ollama)**
* Новый `reference/protocol/executor_detection.md`: мета-процедура до генерации плана (шаг 4б алгоритма). Ollama `/api/tags` + `/api/show` (context_length), совместимые API, консервативный дефолт 4096/1024.
* Списки «популярных моделей» намеренно не вшиваются — перечисляется только обнаруженное в среде.
* Источник ограничений фиксируется; при дефолте gate добавляет `EXECUTOR_CONSTRAINTS_ASSUMED`.

**3. Единая сущность шага (аудит П.4 — два источника истины)**
* Упразднены `AIO` и `VERIFY`-чекпоинты как исполняемые единицы задачи.
* Иерархия: `Фаза → Задача → Шаг`. У шага `action` + `done_criteria` (единственный источник истины проверки).
* Интеграционные проверки выживают как `task_verifications` / `phase_verifications` (охватывают несколько шагов, не дублируют done_criteria).
* Поле `verify_checkpoints` контракта удалено.

**4. Атомарность обязательна всегда (уточнение пользователя + аудит П.10)**
* Устранено противоречие обязательности: декомпозиция обязательна для всех планов (§10), «Блок A» как условный профиль упразднён, его содержимое интегрировано в ядро (§10–§12).
* `granularity` — не переключатель вкл/выкл, а набор порогов из constraints (`atomic`/`micro`; `standard` упразднён).
* Новый принцип §2.7: максимальная атомарность = запрет шагов, нарушающих критерии, а не принудительное дробление тривиальных задач (закрывает претензию «адаптивность к простым задачам 5/10»).

**5. Единая иерархия retry + error_class (аудит П.5)**
* Иерархия: глобальный лимит → задача → шаг; переопределение только в сторону уменьшения.
* Таксономия `error_class` (10 классов) с политикой по каждому: `TRANSIENT`/`OUTPUT_PARSE_FAILURE`/`VERIFICATION_FAILURE` — ретрай; `PERMISSION_DENIED`/`FORBIDDEN_COMMAND`/`UNSAFE_ACTION`/`VERIFICATION_LOGIC_FAILURE`/`ROLLBACK_UNAVAILABLE`/`MISSING_INPUT` — без ретрая.
* Step-retry никогда не отменяет `do_not_retry_on` задачи.

**6. Матрица областей rollback + rollback_mode (аудит П.6)**
* Четыре уровня: шаг → задача → фаза → среда; всегда первым выполняется самый локальный применимый.
* Таксономия `rollback_mode`: `REVERSE_ACTION` / `RESTORE_SNAPSHOT` / `RESTORE_FROM_ARTIFACT` / `COMPENSATING_ACTION` / `MANUAL` / `NONE`.
* Дефолты по типу действия: CREATE→REVERSE_ACTION, EDIT/APPEND→RESTORE_FROM_ARTIFACT, DELETE→обязателен снимок.

**7. Исправлены форматы вывода (аудит П.7 — баг безопасности)**
* Нормативная таблица action→format: `APPEND_FILE`→`APPEND_PAYLOAD` (оркестратор сам дописывает; исполнитель не получает и не возвращает существующее содержимое), `EDIT_FILE`→`UNIFIED_DIFF` + `unchanged_regions_hash`.
* Запрещённые сочетания зафиксированы: `APPEND_FILE`→`FULL_FILE`, `EDIT_FILE`→`FULL_FILE` без хеша.

**8. Пример исправлен (аудит П.8)**
* Дубль `cargo check` устранён: у AS-0043 — локальный grep, компиляция — отдельный шаг AS-0044 (`origin: EXPLICIT`, строка 91 источника).

**9. Проверочные пакеты разрешены (аудит П.9)**
* `verification_batch` для read-only проверок: без мутаций, без решений исполнителя, один детерминированный инструмент, структурированный per-check результат.

**10. Модульность (аудит П.2, П.3)**
* Ядро `SKILL.md` — 1067 строк (~5 000 слов, ~11–12K токенов) против 2588 строк v5.0 (−59%).
* Runtime-протокол вынесен в `reference/protocol/execution.md`; ядро содержит только контракт «что план обязан передать оркестратору» (§16).
* Схемы и примеры — в `reference/schemas/` и `reference/examples/`, загружаются по требованию.
* Переименование артефакта gate: `small_model_readiness.yaml` → `executor_readiness.yaml`.

### Миграция с v5.0

* `executor.executor_class` + `model.*` → `executor.constraints` (значения возьмите из старого `model.context_tokens`/`max_output_tokens`) или `detection: auto`.
* `granularity: standard` → недопустимо; планы обязаны быть атомарными (`atomic` или `micro`).
* `small_model_readiness.yaml` → `executor_readiness.yaml`.
* Чекпоинты `AIO` → шаги `RUN_COMMAND`; `VERIFY` чекпоинты → `done_criteria` шагов или `task_verifications`/`phase_verifications`.
* Поле `verify_checkpoints` в контрактах шагов — удалить.
