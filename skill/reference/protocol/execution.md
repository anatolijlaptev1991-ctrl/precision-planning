# Протокол исполнения плана оркестратором

**Файл:** `reference/protocol/execution.md`
**Назначение:** полный runtime-протокол для оркестратора. Загружается оркестратором, а не планировщиком. В ядре `SKILL.md` остаётся только контракт «что план обязан передать оркестратору» (§16 ядра).
**Граница ответственности:** планировщик генерирует план и контракты; оркестратор исполняет. Этот файл описывает только исполнение.

---

## 1. Роли

### 1.1. Оркестратор

Оркестратор:

1. Читает `plan_hierarchy.yaml`.
2. Проверяет `environment_profile.resolved.yaml` и `executor_profile.resolved.yaml`.
3. Проверяет совместимость плана со средой и исполнителем.
4. Запускает фазы в допустимом порядке.
5. Оркестрирует шаги по протоколу §4.
6. Обрабатывает деградацию, блокеры, регенерации и аудит.
7. Формирует финальный отчёт.

### 1.2. Агент фазы

1. Читает `phase_context.yaml`.
2. Проверяет rollback-состояние.
3. Планирует задачи с учётом ресурсных блокировок.
4. Запускает агентов задач.
5. Собирает evidence.
6. Проверяет локальный частичный порядок.
7. Запускает интеграционные проверки фазы.

### 1.3. Агент задачи

1. Читает `task_NN.yaml`.
2. Валидирует команды по профилю среды.
3. Запускает шаги задачи в допустимом порядке.
4. Собирает evidence.
5. Применяет retry-иерархию §5.
6. Выполняет rollback при необходимости и разрешении.
7. Обновляет статус задачи.

### 1.4. Агент-исполнитель шага

1. Получает ровно один контракт шага (заполненный шаблон промпта).
2. Выполняет ровно одно действие контракта.
3. Отвечает строго по `output_contract`, начиная ответ с маркера `STEP_OK` или `STEP_BLOCKED`.
4. Не использует знания вне `knowledge_boundary` контракта.
5. При нехватке данных отвечает `STEP_BLOCKED: <чего не хватает>` вместо догадок.

## 2. Цикл выполнения фазы

```text
Для каждой фазы:
    если фаза пуста:
        status = EMPTY
        перейти к следующей фазе

    attempts = 0
    пока attempts < max_phase_attempts:
        attempts += 1

        проверить rollback_available
        если rollback требуется и require_rollback_confirmation:
            запросить подтверждение
            если подтверждение недоступно:
                degradation_level = PLAN_WITH_BLOCKERS
                приостановить выполнение

        восстановить состояние до фазы

        построить расписание задач:
            - конфликтующие ресурсы последовательно
            - POTENTIAL_RESOURCE_CONFLICT последовательно
            - POTENTIALLY_ATOMIC без параллелизма
            - независимые задачи параллелить до max_parallel_tasks

        выполнить задачи

        если пользователь пропустил задачу:
            если regeneration_count < max_regenerations_per_plan:
                немедленно регенерировать хвост плана
                regeneration_count += 1
                продолжить с обновлённой структурой
            иначе:
                degradation_level = PLAN_WITH_BLOCKERS
                status = BLOCKED_REGENERATION_LIMIT_EXHAUSTED
                приостановить выполнение до ручного разрешения

        проверить частичный порядок
        если выверка не прошла:
            если attempts == 5: запросить пользователя
            продолжить

        выполнить интеграционные проверки фазы
        если проверки успешны:
            status = success
            перейти к следующей фазе

    если attempts >= max_phase_attempts:
        status = FATAL
        сохранить частичные результаты
        остановить выполнение
```

## 3. Поведение при исчерпании лимита регенераций

```yaml
regeneration_limit_exhausted:
  degradation_level: "PLAN_WITH_BLOCKERS"
  current_phase_status: "BLOCKED_REGENERATION_LIMIT_EXHAUSTED"
  execution: "suspended"
  required_action: "manual_resolution"
```

Правила: текущая фаза не считается успешно завершённой; дальнейшее выполнение приостанавливается; финальный отчёт указывает пропущенную задачу, исключённые зависимые элементы и причину; автоматическое продолжение запрещено.

## 4. Протокол исполнения атомарного шага

```text
1. Проверить предусловия: все зависимости шага в статусе success;
   хеши inputs совпадают с реестром; для EDIT_FILE/APPEND_FILE хеш
   целевого файла совпадает с context_pack.
   При расхождении — пересобрать контекст шага, зафиксировать дрейф.

2. Собрать промпт по шаблону (reference/schemas/atomic_step.md, §3).
   Проверить C5 (бюджет токенов).

3. Вызвать исполнителя ровно один раз.

4. Распарсить вывод строго по output_contract.
   Допустима только детерминированная нормализация (удаление
   ограждающих код-заборов, обрезка текста вне маркеров) с записью
   события в журнал. Интерпретировать невалидный вывод «по смыслу»
   запрещено: это failure attempt с error_class = OUTPUT_PARSE_FAILURE.

5. Применить результат:
   - FULL_FILE      → атомарная запись файла (temp + rename)
   - UNIFIED_DIFF   → применить патч; проверить unchanged_regions_hash
   - APPEND_PAYLOAD → оркестратор сам дописывает payload в конец файла
   - COMMAND_OUTPUT → выполнить argv через command policy
   Все записи — через path policy, с фильтрацией секретов.

6. Выполнить все done_criteria машинно. Собрать evidence.

7. Все критерии пройдены → status: success → опубликовать артефакты
   в реестре → перейти к следующему шагу.

8. Не пройдены → retry по иерархии §5. При исчерпании:
   status: blocked → escalation:
   user | escalate_constraints (тот же контракт исполняется
   исполнителем с более широкими ограничениями) | block.
   Шаги, зависимые от заблокированного, не запускаются.

9. Каждые drift_check_interval шагов — сверка состояния:
   существование целевых файлов, хеши артефактов, статусы.
```

Оркестратору запрещено: склеивать несколько шагов в один вызов; пропускать `done_criteria`; продвигать зависимые шаги после блокировки; изменять вывод исполнителя сверх детерминированной нормализации.

### 4.1. Проверочные пакеты (batch verification)

Запрет пакетирования (ядро §10.3, R4) касается **мутирующих** шагов. Для проверок без побочных эффектов допускается объединение в один вызов инструмента при одновременном выполнении условий:

1. Все объединяемые операции — read-only (`READ_FILE`, `ASSERT_STATE`, grep, проверки существования).
2. Ни одна не требует решения исполнителя (только детерминированное выполнение).
3. Выполняются одним инструментом за один вызов (один скрипт-валидатор, одна команда).
4. Результат структурирован (JSON/YAML с per-check статусами).

Оформление: `verification_batch` внутри задачи, не шаг. Каждый элемент пакета сохраняет собственный `id` для трассировки. Мутирующие действия в пакет включать запрещено.

## 5. Иерархия retry

Единственный источник истины. Переопределение разрешено только в сторону **уменьшения** лимитов.

```text
Глобальный лимит  (skill_config.max_task_attempts, max_phase_attempts)
    ↓ не может быть превышен
Лимит задачи      (task.retry_strategy.max_attempts)
    ↓ не может быть превышен
Лимит шага        (step.retry_policy.max_attempts, ≤ executor.step_retry_max)
```

### 5.1. Классификация ошибок

Каждый failure attempt обязан получить `error_class`:

```yaml
error_class:
  enum:
    - TRANSIENT                 # сетевой сбой, таймаут, временная недоступность
    - OUTPUT_PARSE_FAILURE      # вывод не соответствует output_contract
    - VERIFICATION_FAILURE      # done_criteria не пройдены
    - VERIFICATION_LOGIC_FAILURE# done_criteria противоречивы или неисполнимы
    - PERMISSION_DENIED
    - FORBIDDEN_COMMAND
    - UNSAFE_ACTION
    - ROLLBACK_UNAVAILABLE
    - RESOURCE_CONFLICT
    - MISSING_INPUT             # нехватка данных контракта
```

### 5.2. Политика по классам

| error_class | Поведение |
|---|---|
| `TRANSIENT` | Ретрай с exponential backoff, до лимита шага |
| `OUTPUT_PARSE_FAILURE` | Ретрай с error-feedback (секция `# ОШИБКА ПРЕДЫДУЩЕЙ ПОПЫТКИ`), до лимита шага |
| `VERIFICATION_FAILURE` | Ретрай с error-feedback и текстом непройденной проверки |
| `VERIFICATION_LOGIC_FAILURE` | **Без ретрая.** Блокер плана: дефект контракта |
| `PERMISSION_DENIED` | **Без ретрая.** Блокер; в интерактивном режиме — запрос пользователя |
| `FORBIDDEN_COMMAND` | **Без ретрая.** Блокер безопасности |
| `UNSAFE_ACTION` | **Без ретрая.** Немедленный стоп шага; аудит |
| `ROLLBACK_UNAVAILABLE` | **Без ретрая.** `PLAN_WITH_BLOCKERS` |
| `RESOURCE_CONFLICT` | Сериализация задач; ретрай после освобождения ресурса |
| `MISSING_INPUT` | **Без ретрая.** Шаг отвечает `STEP_BLOCKED`; пересборка контракта планировщиком |

### 5.3. Правила

1. Step-retry применим только к ошибкам вывода/верификации исполнителя и **никогда** не отменяет `do_not_retry_on` задачи.
2. Задача может сузить, но не расширить список `do_not_retry_on` шага.
3. Каждый ретрай — событие в журнале с `error_class`, `attempt`, оставшимся бюджетом попыток.
4. Пятая последовательная неудача на уровне задачи: интерактивный режим — запрос пользователя; автономный — `BLOCKED`.

## 6. Матрица областей rollback

Rollback существует на четырёх уровнях. **Всегда выполняется самый локальный применимый откат первым.**

| Уровень | Область | Когда применяется | rollback_mode |
|---|---|---|---|
| Шаг | Откат текущего шага | Провал шага после применения результата | `REVERSE_ACTION` / `RESTORE_FROM_ARTIFACT` |
| Задача | Возврат к состоянию перед задачей | Провал задачи, блокировка критического шага | `REVERSE_ACTION` / `RESTORE_SNAPSHOT` / `COMPENSATING_ACTION` |
| Фаза | Аварийное восстановление всей фазы | `FATAL` фазы, нарушение инвариантов | `RESTORE_SNAPSHOT` |
| Среда | Инфраструктурный snapshot | Страховочный уровень перед фазой | `RESTORE_SNAPSHOT` |

### 6.1. Таксономия rollback_mode

```yaml
rollback_mode:
  enum:
    - REVERSE_ACTION          # обратное действие (delete созданного, restore изменённого)
    - RESTORE_SNAPSHOT        # восстановление snapshot среды/файловой системы
    - RESTORE_FROM_ARTIFACT   # восстановление содержимого из артефакта реестра
    - COMPENSATING_ACTION     # компенсирующая операция (когда обратного действия нет)
    - MANUAL                  # только ручной откат
    - NONE                    # откат невозможен (обязана пометка NOT_ROLLBACKABLE)
```

### 6.2. Правила приоритета

1. Провал шага → откат шага. Откат задачи не инициируется, пока не исчерпан step-retry.
2. Провал отката шага → эскалация на откат задачи → фазы. Каждая эскалация — событие CRITICAL в журнале.
3. Шаг с `rollback_mode: NONE` допустим только с `origin: EXPLICIT` или `USER_CONFIRMED` и тегом `DESTRUCTIVE_IRREVERSIBLE`.
4. `require_rollback_confirmation` профиля среды имеет приоритет над автоматикой всех уровней (ядро §5.3).
5. Для `EDIT_FILE`/`APPEND_FILE` rollback шага по умолчанию — `RESTORE_FROM_ARTIFACT` (снимок до шага в реестре); для `CREATE_FILE` — `REVERSE_ACTION` (удаление); для `DELETE_FILE` — обязателен артефакт-снимок до удаления.
